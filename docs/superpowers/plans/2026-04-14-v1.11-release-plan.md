# v1.11.0 Release Plan — Tahoe Compatibility & Reliability

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix the 500 errors and broken functionality on macOS 26 Tahoe, reduce error noise from gateway restarts, and unblock headless operation.

**Architecture:** These are targeted fixes to the existing BlueBubbles server codebase. No new subsystems — code changes in AppleScript builder, database queries, webhook service, and window initialization.

**Tech Stack:** TypeScript, Vitest, Koa, TypeORM, better-sqlite3

**Spec:** `docs/research/2026-04-14-private-api-tahoe.md` + `docs/research/2026-04-14-headless-operation.md`

**Context:** Research identified five separate Tahoe breakages. Three are fixable with code changes. Two (DYLIB injection, imagent XPC entitlements) are architectural blockers with no known bypass.

## Execution Order

### Phase 1: Fix Tahoe Send/Read Failures (Critical — fixes 500 errors)

These two issues are the root cause of agent confusion and reinforced workaround instructions.

#### Issue #18: Map `any` service type to `iMessage` in AppleScript sends

**Problem:** macOS 26 changed chat GUIDs from `iMessage;-;phone` to `any;-;phone`. AppleScript rejects `any` with error -1700, causing silent send failures (500 errors).

**Files to investigate:**
- `packages/server/src/server/api/apple/scripts.ts` — find where service type is extracted from chat GUID and used in AppleScript construction
- Search for `buildServiceScript`, `service`, `iMessage;-;`, `buddy` in the apple scripts module

**Fix:** Before constructing the AppleScript, map `any` → `iMessage` in the service type. Preserve `SMS` and `RCS` pass-through.

**Tests:**
- `any` service type maps to `iMessage`
- `iMessage`, `SMS`, `RCS` pass through unchanged
- Constructed AppleScript uses the mapped value

#### Issue #19: Read message content from `attributedBody` column

**Problem:** macOS 26 sets the `text` column to NULL. Content is only in `attributedBody` (NSAttributedString binary blob). Message detection breaks.

**Files to investigate:**
- `packages/server/src/server/databases/imessage/entity/Message.ts` — check `universalText()` method for existing `attributedBody` fallback logic
- `packages/server/src/server/databases/imessage/entity/` — check for any `attributedBody` handling
- Search for `attributedBody`, `NSAttributedString`, `text` column queries

**Fix:** Ensure `universalText()` falls back to decoding `attributedBody` when `text` is NULL. The NSAttributedString blob needs to be parsed to extract plain text (it's a binary plist format).

**Tests:**
- Message with NULL text but populated attributedBody returns correct text
- Message with populated text (pre-Tahoe) still works
- Empty attributedBody returns empty string

### Phase 2: Reliability Improvements

#### Issue #14: Headless mode null window crash

**Problem:** `preChecks()` in `packages/server/src/server/index.ts` line 795 calls `this.window.minimize()` without null guard. Crashes when headless mode skips window creation.

**Fix:** `if (this.window) this.window.minimize();`

**Tests:**
- preChecks() with null window doesn't crash
- preChecks() with valid window still calls minimize()

#### Issue #15: Webhook retry with exponential backoff

**Problem:** `sendPost()` in `packages/server/src/server/services/webhookService/index.ts` fires and forgets. Failed webhooks are silently dropped. Gateway restarts cause a burst of lost messages and noisy errors.

**Fix:** Add retry with exponential backoff (3 attempts: 1s, 2s, 4s). After max retries, log as warning (not error).

**Tests:**
- Retries on connection refused
- Succeeds on retry after initial failure
- Gives up after max retries with warning log
- Successful delivery is not retried

### Phase 3: Research & Prototype (Separate Branch)

#### Issue #20: Accessibility API typing indicators

**Problem:** Private API typing indicators are dead on Tahoe (imagent XPC entitlement wall). The Accessibility API (`AXUIElement`) may offer an injection-free alternative by focusing Messages.app's text input field.

**Research tasks:**
- [ ] Prototype AXUIElement query to find Messages.app text input
- [ ] Test if focusing the input triggers typing indicator on remote end
- [ ] Measure performance of AX query + focus cycle
- [ ] Document Accessibility permission requirements
- [ ] Determine if this can work without bringing Messages.app to foreground

**This is R&D — may not result in a shippable fix.**
